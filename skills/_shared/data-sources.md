# Описание файла

Общий справочник по антифрод-источникам данных. Содержит смысловые блоки: назначение hit/raw таблиц, выбор источника, ключи связывания и базовые группы полей.

## Главные источники

| Источник | Назначение | Ограничение |
|---|---|---|
| `cspfs_repo_features3.hits_extra_info_129372427_view` | Сработки антифрода, решения, резолюции, save-флаги, краткий контекст до/после. | Не является полной историей клиента. |
| `csp_afpc_sss_inc.cards_event` | Полная история карточного канала: POS, e-commerce, ATM, авторизации, токены. | Строка не означает сработку. |
| `csp_afpc_sss_inc.uko_event` | Полная история не карточного контура: ДБО, СБП, ВСП, переводы, сервисные операции. | Не подменяет карточный контур. |
| `cspfs_repo_features3.cards_event` | FS-алиас карточной raw-истории, если проект использует новый контур. | Проверяй фактическую схему проекта. |

## Выбор источника

- Сначала ищи сработку в hits по `event_id`.
- Для поведения клиента до/после сработки переходи в raw-историю: `cards_event` или `uko_event`.
- Выбор raw-таблицы делай по `event_channel`, `surface`, `product`, `type_operation` и смыслу операции.
- Если `event_id` не связывает таблицы напрямую, используй `epk_id` или `user_id`, дату/время, сумму, тип операции, получателя/ТСТ и реквизиты.

## Базовые поля hits

- Идентификация: `event_id`, `epk_id`, `user_id`, `event_dt`, `event_time`.
- Канал и тип: `event_channel`, `sub_channel`, `event_type`, `sub_type`, `type_operation`, `purpose`, `surface`, `product`.
- Описание операции: `event_description`. Не используй несуществующие поля `description` или `transaction_description`, если фактическая схема таблицы их не содержит.
- Деньги: `transaction_amount`, `transaction_amount_in_rub`, `client_balance`, `transaction_amount_currency`.
- Антифрод: `policy_action`, `main_rule`, `type_accept`, `source_type_accept`, `resolution_first`, `resolution_last`, `accept_time_sec`.
- KPI и жалобы: `is_save`, `marked_as_not_save_reason`, `has_claim`, `payment_transaction_flag`.
- Контекст: `previous_events`, `posterious_events`, `hits_extra_facts`.

## Хранение дат

- `event_dt` в тестовых Spark-like CSV хранится как `YYYYMMDD`, например `20260124`.
- Для фильтров `read_table` по `event_dt` используй значения в том же формате: `20260101`, `20260131`.
- Не используй ISO-формат `YYYY-MM-DD` для `event_dt`, если схема или данные явно не показывают такой формат.
- Если пользователь говорит "последний доступный месяц", сначала определи максимум `event_dt` в доступных данных, затем используй календарный месяц этого максимального значения.

## Правила фильтрации read_table

- Все фильтры `read_table.filters` применяются последовательно и объединяются через логическое `AND`.
- Не передавай несколько `contains` по разным ключевым словам как замену `OR`: такой запрос потребует совпадения со всеми словами сразу.
- Для поиска по нескольким словам или нескольким полям сначала выгрузи минимальный срез за период через `read_table`, затем выполни `OR`-фильтрацию через Python-анализ.
- `read_table` не выполняет агрегаты, сортировку, `LIKE` или произвольный SQL. Для `max`, `count`, группировок и текстового `OR` используй Python-анализ после выборки.

## Базовые поля cards_event

- Идентификация: `event_id`, `epk_id`, `user_id`, `event_dt`, `event_time`.
- Описание операции: `event_description`.
- Merchant/MCC: `atm_merchant_name`, `atm_merchant_id`, `atm_mcc`, `atm_mcc_name`.
- Деньги: `transaction_amount`, `transaction_amount_in_rub`, `transaction_amount_currency`.

## Raw-события

Raw-таблицы нужны для полной последовательности действий клиента, включая операции без антифрод-сработки. Для дневного разбора выгружай события за день сработки и релевантные дни из исторического окна.

## Связывание

Приоритет ключей:

1. `event_id` в обеих таблицах.
2. `epk_id`/`user_id` + `event_dt` + `event_time`.
3. Клиент + временное окно + сумма + тип операции + получатель/merchant/реквизиты.

## Осторожность

- Не делай вывод о мошенничестве только по факту наличия строки в hits.
- Не считай raw-событие остановленной операцией без проверки hits.
- Не показывай чувствительные поля без маскирования; список таких полей см. в `antifraud-core.md`.
