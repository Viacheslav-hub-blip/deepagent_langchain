# Расследование паттерна (опирается на `antifraud-hit-table` + raw events)

## 1. Зафиксировать кейс
- Есть **event_id** → запись сработки из hits; извлечь `epk_id`, `event_dt`, канал, сумму, merchant/получателя, тип, `main_rule`, решение.
- Есть **epk_id + дата** → сработки клиента за эту дату, выбрать релевантную.
Без `epk_id` / `event_dt` (и контекста) нельзя строить окна и сравнения.

## 2. История сработок
Окно: **180 дней до `event_dt` включительно до текущей сработки** (как в политике выгрузки). Считать частоту, каналы, правила, суммы, повторы merchant/получателей; сравнить текущую сработку с прошлыми.

## 3. События по дням
Для **дня текущей сработки** и для **дней с прошлыми сработками** в окне — полные события из `cards_event` или `uko_event` (выбор по каналу из сработки, см. skills карточек/UKO). Смотреть до/после времени сработки, серии, смену канала/гео/получателя.

## 4. Похожие операции
Жёстко: тот же merchant или получатель + близкая сумма + канал. Потом ослаблять критерии (сумма/получатель/тип/сценарий дня).

## 5. Два уровня вывода
- **Долгосрочно (180 д):** повторы сумм, контрагентов, каналов, правил, «тихие» успешные операции.
- **Внутри дня:** совпадает ли сценарий дня с прошлыми днями со сработками.

## 6. Порядок шагов (один список)
1. Сработка из hits → ключи и канал.  
2. Сработки клиента за 180 дней до `event_dt`.  
3. События за каждый нужный день в правильной raw-таблице.  
4. Похожие транзакции и сравнение дней.  
5. Вывод: типичное vs выбивающееся поведение и что могло привести к сработке (только по данным).
