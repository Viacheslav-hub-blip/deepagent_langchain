# Поиск записей по смысловому описанию

## Назначение

Используй этот skill, когда пользователь просит найти записи, события, операции, транзакции или сработки по описанию на естественном языке.

Примеры пользовательских запросов:

* «Найди сработки, связанные с оплатой доставки за январь 2025 года»
* «Посчитай операции, похожие на оплату такси»
* «Выгрузи записи, относящиеся к подпискам»
* «Найди события по оплате маркетплейсов»
* «Сколько было операций, связанных с доставкой еды»
* «Покажи записи, которые по смыслу похожи на регулярные платежи»

Главная особенность задачи: поле `event_description` может быть заполнено разными формулировками. Поэтому задачу нельзя решать через поиск подстроки, проверку вхождения слов или фильтр по ключевым словам в `event_description`.

Правильный подход:

1. Получить все уникальные значения `event_description` за указанный период.
2. Для каждого уникального описания определить связанные коды операций и `mcc_code`.
3. По смыслу выбрать описания, коды операций и `mcc_code`, которые относятся к категории из запроса пользователя.
4. Сделать полную выгрузку записей по выбранным структурированным признакам: кодам операций, `mcc_code`, а при необходимости — по подтвержденным merchant-признакам.
5. Посчитать количество записей и вернуть пользователю проверяемый результат.

## Главное правило

Не ищи записи по прямому вхождению слов в `event_description`.

Неправильный подход:

```text
event_description содержит "доставка"
event_description содержит "delivery"
merchant_name содержит "доставка"
```

Почему это неправильно:

* одно и то же действие может иметь разные описания;
* доставка может быть закодирована через код операции или `mcc_code`, даже если слова «доставка» нет в описании;
* названия мерчантов могут быть сокращенными, транслитерированными, неполными или пустыми;
* поиск по словам может пропустить релевантные записи;
* поиск по словам может добавить нерелевантные записи.

Правильный подход:

```text
период
→ справочник уникальных event_description + коды операций + mcc_code + merchant-признаки
→ семантический выбор релевантных описаний и кодов
→ полная выгрузка по выбранным структурированным признакам
→ подсчет количества
→ ответ с критериями, ограничениями и источниками
```

## Входные параметры из запроса пользователя

Из запроса пользователя нужно выделить:

* смысловую категорию поиска, например: «оплата доставки»;
* период, например: «январь 2025 года»;
* ожидаемый формат результата:

  * только количество;
  * список записей;
  * CSV/XLSX-файл;
  * аналитическая сводка;
* дополнительные фильтры, если они есть:

  * клиент;
  * пользователь;
  * канал;
  * продукт;
  * мерчант;
  * город или регион;
  * статус сработки;
  * статус операции;
  * действие антифрода.

Если период не указан, задай уточняющий вопрос.

Если смысловая категория не указана или слишком расплывчата, задай уточняющий вопрос.

## Обязательные поля

Когда поля доступны в источнике, используй:

* идентификатор записи: `event_id`, `operation_id`, `id` или другой подтвержденный первичный ключ;
* дата и время события: `event_dt`, `operation_dt`, `created_at` или другое подтвержденное поле времени;
* описание события: `event_description`;
* код операции или события: `operation_code`, `event_code`, `op_code`, `event_type` или другое подтвержденное поле кода;
* MCC-код: `mcc_code`;
* merchant-поля: `merchant_name`, `merchant_id`, `merchant_info`, `recipient_info` или похожие подтвержденные поля;
* сумма операции: `amount`;
* статусные поля, если они нужны для задачи: `policy_action`, `resolution`, `status`;
* клиентский ключ, если запрос относится к конкретному клиенту.

Используй только реальные названия полей из preloaded skills, domain context или schema. Не придумывай поля.
Перед использованием любого поля проверь, что оно есть в выбранной таблице по preloaded context, skill-описанию или фактической schema/tool output.
Если поле отсутствует в выбранной таблице, не включай его в select, group_by, filters, aggregations и критерии отбора.

Для таблицы сработок `cspfs_repo_features3.hits_extra_info_129372427_view` не запрашивай `mcc_code` или `atm_mcc`, если этих полей нет в схеме.
Для hits используй только подтвержденные поля таблицы сработок: например, `event_description`, `event_type`, `sub_type`, `type_operation`, `main_rule`, `atm_merchant_name`, `merchant_info`, `pos_info`, `product`, `product_type`, `purpose` и статусные поля, если они есть в schema.

## Общий workflow

### Шаг 1. Определить источник и нужные поля

Используй preloaded domain context и skills, чтобы определить таблицу, где есть:

* `event_description`;
* код операции или события;
* `mcc_code`, если это поле есть в выбранной таблице;
* дата/время события;
* идентификатор записи;
* merchant-поля;
* сумма операции.

Если есть несколько таблиц-кандидатов, выполни конкретную проверку схемы через `data-retrieval-agent`.

Проверка должна вернуть:

* названия таблиц-кандидатов;
* доступные поля описания;
* доступные поля кодов операций;
* доступные MCC-поля, если они есть в выбранной таблице;
* доступные поля даты;
* доступные идентификаторы записей;
* доступные merchant-поля;
* ограничения источников.

Перед формированием задания на чтение данных проверь наличие каждого поля в выбранной таблице. Для hits не добавляй `mcc_code`/`atm_mcc` в задание, если schema таблицы сработок их не содержит.

Не создавай пункт плана вида «изучить источники данных». Это внутренняя подготовка. Если нужна проверка схемы, формулируй ее как конкретную задачу для чтения данных.

### Шаг 2. Построить справочник описаний и кодов за период

Через `data-retrieval-agent` получи справочник уникальных комбинаций за указанный период.

Минимально нужные поля:

* `event_description`;
* код операции или события;
* `mcc_code`, если это поле есть в выбранной таблице;
* merchant-поле, если оно есть;
* количество записей по комбинации, если инструмент умеет агрегацию;
* подтверждение границ периода.

Если инструмент чтения данных не умеет делать агрегацию или `distinct`, получи широкую, но ограниченную по периоду выгрузку только с нужными полями и передай ее в `python-analysis-agent` для дедупликации и подсчета.

Цель шага — загрузить в контекст реальное разнообразие `event_description` и понять, какие коды операций и `mcc_code` с ними связаны.

### Шаг 3. Семантически выбрать релевантные коды операций и MCC

Проанализируй справочник из шага 2 по смыслу.

Выбери значения, которые относятся к категории из запроса пользователя не по вхождению слов, а по смыслу.

Для каждого выбранного кандидата зафиксируй:

* выбранный `event_description`;
* код операции или события;
* `mcc_code`, если поле доступно в выбранной таблице;
* merchant-признаки;
* количество записей в справочнике;
* почему кандидат относится к нужной смысловой категории;
* уровень уверенности:

  * высокая уверенность;
  * средняя уверенность;
  * низкая уверенность.

Исключай кандидатов, которые подходят только из-за слабого совпадения по словам.

Если связь неоднозначна, отдели кандидаты с высокой уверенностью от кандидатов со средней или низкой уверенностью. Не смешивай их в одном итоговом количестве без пояснений.

### Шаг 4. Сделать полную выгрузку по выбранным кодам

После семантического выбора сделай полную выгрузку записей за период по выбранным структурированным признакам:

* кодам операций или событий;
* `mcc_code`, если поле доступно в выбранной таблице;
* merchant-признакам, если они нужны и подтверждены;
* дополнительным фильтрам пользователя, если они есть.

Основной фильтр должен строиться по подтвержденным кодам и структурированным полям, а не по поиску подстроки в `event_description`.

В полной выгрузке нужны поля:

* идентификатор записи;
* дата и время события;
* `event_description`;
* код операции или события;
* `mcc_code`, если поле доступно в выбранной таблице;
* merchant-поля;
* сумма;
* статусные поля, если применимо;
* клиентский ключ, если применимо.

Если результат большой, сохрани полную выгрузку в файл.

### Шаг 5. Посчитать количество и проверить результат

Передай полную выгрузку в `python-analysis-agent`.

Нужно посчитать:

* общее количество записей;
* количество по коду операции или события;
* количество по `mcc_code`;
* количество по мерчанту, если merchant-поле есть;
* количество по дням, неделям или месяцам внутри периода, если это полезно;
* общую сумму, если поле `amount` доступно и релевантно;
* количество записей, исключенных из-за низкой уверенности или отсутствующих кодов.

Обязательные проверки:

* границы периода применены корректно;
* выбранные коды соответствуют семантическому сопоставлению из шага 3;
* основной отбор не был сделан через substring-поиск по `event_description`;
* пустой результат возвращается как `0 rows`;
* preview/sample не выдается за полный dataset.

### Шаг 6. Сохранить файл, если он нужен

Если пользователь просит файл или результат слишком большой для текстового ответа, сохрани итоговые записи через `save_analysis_file`.

Файл должен содержать:

* все отобранные записи;
* смысловую категорию отбора;
* период;
* код операции или события;
* `mcc_code`, если поле доступно в выбранной таблице;
* merchant-поля;
* сумму;
* поля, на основании которых выполнялся отбор.

Возвращай абсолютный путь к файлу только если `save_analysis_file` вернул успешный статус и `absolute_path`.

### Требования к финальному ответу

Финальный ответ supervisor-а должен содержать:

* краткий вывод;
* смысловую категорию поиска;
* период;
* итоговое количество записей;
* выбранные коды операций или событий;
* выбранные значения `mcc_code`;
* ключевых мерчантов, если они доступны;
* использованные таблицы и поля;
* метод отбора;
* явное указание, что поиск по вхождению слов не использовался как основной критерий;
* ограничения и неоднозначные случаи;
* путь к файлу, если файл был создан.

## Правила семантического отбора

Используй смысловой выбор на основе полного справочника описаний и кодов за период.

Кандидат можно включить, если выполняется хотя бы одно условие:

* `event_description` явно описывает нужную смысловую категорию;
* код операции устойчиво связан с этой категорией;
* `mcc_code` в данных связан с этой категорией;
* merchant-признаки подтверждают категорию;
* несколько описаний с одним кодом/MCC указывают на одну и ту же категорию.

Кандидата нужно пометить как неоднозначного, если:

* описание слишком общее;
* `mcc_code` может относиться к нескольким категориям;
* merchant_name отсутствует;
* код операции смешивает разные типы событий;
* есть только один слабый текстовый признак.

Неоднозначные кандидаты нельзя молча включать в основной счетчик. Их нужно либо исключить, либо показать отдельно.

## Антипаттерны

Не делай так:

```text
description contains "доставка"
description contains "delivery"
merchant_name contains "еда"
```

Запрещено:

* решать задачу поиском по ключевым словам;
* выбирать записи до просмотра справочника уникальных `event_description`;
* игнорировать коды операций;
* игнорировать `mcc_code`, если поле доступно в выбранной таблице;
* считать preview полным dataset;
* придумывать названия таблиц и полей;
* возвращать только примеры, если пользователь просил количество;
* скрывать неоднозначные совпадения;
* делать итоговый вывод без таблиц, полей, фильтров и периода.

## Обработка пустого результата

Если релевантные записи не найдены, верни:

```text
Найдено 0 записей.

Период: <period>.
Проверенная таблица: <table_name>.
Проверенные поля: <fields>.
Метод: сначала был построен справочник event_description → код операции → доступные структурированные признаки, затем были проверены структурированные коды.
Ограничения: <limitations>.
```

Не утверждай, что записей нет во всей системе, если не был проверен полный релевантный dataset.

## Пример

Пользовательский запрос:

```text
Найди сработки, связанные с оплатой доставки за январь 2025 года.
```

Правильный workflow:

1. Определить период: январь 2025 года.
2. Определить смысловую категорию: оплата доставки.
3. Проверить наличие нужных полей в выбранной таблице.
4. Получить уникальные значения `event_description` за январь 2025 года вместе с кодом операции, доступными структурированными признаками, merchant-полями и количеством записей.
5. Семантически определить, какие описания, коды операций и доступные структурированные признаки соответствуют оплате доставки.
6. Сделать полную выгрузку записей за январь 2025 года по выбранным кодам операций и доступным структурированным признакам.
